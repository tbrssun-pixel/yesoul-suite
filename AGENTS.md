Use subagents when they are available, permitted by the current tool/session instructions, and materially improve quality, speed, or risk control.

Prefer subagents for independent quality review, QA, security review, UX review, architecture review, checking hypotheses or plans from a skeptical second perspective, parallel exploration of clearly separate questions, or implementation subtasks with disjoint file ownership and low merge-conflict risk.

Do not use subagents for simple linear tasks, urgent blocking work, vague delegation, or work that the main agent must immediately do to stay on the critical path.

Before delegating, define a narrow mission, inputs, allowed scope, expected output, and stop condition. The main agent remains responsible for integrating results, resolving disagreements, verifying evidence, and producing the final answer.

## Deploy Policy

This project currently produces a debug Android APK for manual installation and has a release AAB helper for future Google Play controlled beta validation. There is no public web deploy target.

Live release signing, Play Store publication, external distribution, signing key changes, production credentials, and CI secret changes require a separate explicit user request. Release signing is configured only through `OWL_BIKE_RELEASE_*` environment variables. Use `docs/deploy.md` for assessment, safe checks, controlled beta gates, health check, and rollback notes.
